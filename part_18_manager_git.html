1. Менеджери проектів.

Менеджер проектів - це програма чи хмарний сервіс, який забезпечує комунікацію в команді, дозволяє створювати задачі, відстежувати роботу кожного учасника команди, створювати базу знань по проектам і т.п.

- Trello.

2. Git

Git - система керування контентом.

Окрім гіта є ще декілька, наприклад, Turtley SVN.

Git - це технологія, якою користуються сервіси типу GitHub, BitBucket та інші.

Зареєструйтесь на GitHub та встановіть програму git (нам потрібна буде лише її консольна версія).

Основні команди:

// для навігації в windows:
dir - роздрукувати на екран вміст поточної папки
d: - перейти на диск d:
cd 2016-12 - перейти в папку
cd .. - перейти в папку вище
cd ../../futbolki/images
// для навігації в Linux / MacOS замість dir використовується:
ls -l
mkdir назва_папки - створити папку в поточному каталозі

// git
git clone посилання - клонувати репозиторій з сервера на локальну машину.
git status - перевірити поточний статус поточної гілки.
git checkout -b нова_гілка - створити нову гілку.
git checkout гілка - перейти на іншу гілку (спочатку створіть коміт в цій, якщо були зміни).
git add . - добавити всі файли в коміт (точку збереження).
git add test.html - добавити в коміт тільки один конкретний файл.
git commit -m "текст-опис" - створити коміт (точку збереження) поточної гілки.
git push origin гілка - закинути коміт на сервер.
git pull origin гілка - зтягнути оновлення з сервера.
git merge гіка - об'єднати поточну гілку зі вказаною.
git reset --hard HEAD - якщо щось сталося і все дуже погано [картинка з ядерним вибухом]
git cherry-pick комміт - дозволяє вибрати комміт, який можна запушити на сервер

Основи розгалужень в git.

Алгоритми дій у колективному проекті.

Включаємося в роботу:

запускаємо консоль:
win+r, cmd
переходимо на диск, де у нас робоча папка:
D:
переходимо в робочу папку:
cd work/frontend/2017-02
створюємо папку проекта:
mkdir mebel
клонуємо репозиторій собі локально, створиться папка з репозиторієм:
git clone https://github.com/...
переходимо в папку з репозиторієм:
cd mebel_???? (замість знаків питання підставте потрібне)
створюємо свою гілку і переходимо в неї:
git checkout -b назва_гілки
виконуємо свою частину роботи:
верстаємо свій блок
перевіряємо поточний статус нашої гілки:
git status
добавляємо змінені файли в коміт:
git add . (крапка через пробіл від слова!)
перевіряємо, чи всі файли добавилися:
git status
створюємо коміт, в коментар пишемо те, що ми зробили, наприклад "section #advantages done":
git commit -m "коментар"
під час створення першого коміту гіт може попросити вас ввести дані про себе: логін на гітхаб та email, на який ви реєструвалися. введіть ці дві команди за наведеним прикладом.
заливаємо свою гілку на сервер:
git push origin назва_гілки
переходимо на github.com; якщо з моменту клонування проекту в гілці master не було ніяких змін, то можна створювати запит на об'єднання своєї роботи з гілкою master: знаходимо вкладку Pull Request, нажимаємо кнопку Create Pull Request, в base: залишаємо гілку master, в compare: знаходимо у списку свою гілку, гортаємо сторінку та про всяк випадок перевіряємо свої зміни, нажимаємо кнопку Create Pull Request, потім ще раз таку саму кнопку.
тімлід (старший розробник) перевірить вашу роботу і якщо все гаразд - об'єднає її з гілкою master.
якщо ж у гілці master були зміни - то скористайтесь наступним алгоритмом:

Створюємо запит на додавання своєї роботи в гілку master:

перевіряємо поточний статус нашої гілки (і в якій гілці ми взагалі знаходимося):
git status
якщо є змінені файли - потрібно зробити коміт:
git add .
git commit -m "коментар про те що я зробив"
git status
якщо пише щось типу "nothing to commit" - можна переключатися у іншу гілку:
git checkout master
пробуємо підтягнути оновлення з сервера:
git pull origin master
переключаємося в свою гілку:
git checkout назва_гілки
якщо на сервері оновлень не було - переходьте до пункту заливки гілки на сервер.
якщо на сервері були оновлення - підтягуємо їх у свою гілку:
git merge master
вірогідно, під час об'єднання гілок виникнуть конфлікти, їх треба повиправляти:
виправляємо конфлікти
перевіряємо статус гілки:
git status
добавляємо змінені файли в коміт:
git add . (крапка через пробіл від слова!)
перевіряємо, чи всі файли добавилися:
git status
створюємо коміт, в коментар пишемо те, що ми зробили, наприклад "resolved conflicts in section #advantages":
git commit -m "коментар"
заливаємо свою гілку на сервер:
git push origin назва_гілки
далі йдемо на github.com і створюємо Pull Request як у попередньому алгоритмі.

Домашнє завдання.

Рівень 1.

1. Зареєструйтесь у менеджерах проектів, створіть по проекту в кожній системі, подосліджуйте можливості.

2. Створіть задачі різних типів, позакривайте їх, перегляньте статистику.

3. Створіть проект git в консолі, добавте в нього файли з Колесова. Створіть дві гілки, вносьте зміни в ці гілки.

4. Перейдіть у основну гілку і під'єднайте до неї дві інші гілки.

5. Створіть проект на GitHub, клонуйте його, закиньте в проект файли з Minecraft, внесіть зміни (поміняйте колір заголовка), закиньте на GitHub.

6. Створіть нову гілку, внесіть зміни (поміняйте розмір шрифта бокової панелі), закиньте на GitHub та змержіть з основною гілкою.

7. Створіть дві нові гілки, в обох гілках зробіть зміни в одному й тому ж місці (наприклад, змініть h1 на h2 в одній гілці та на h3 в іншій), спробуйте змержити ці гілки з основною гілкою.

Рівень 2.

1. Створіть нову гілку в проекті Minecraft, вносьте зміни:
   - задайте трохи більший розмір заголовку та синій колір датам в новинах, закиньте на GitHub;
   - задайте підкреслення посиланням головного меню, закиньте на GitHub;
   - поміняйте пару телефонів (і вгорі, і внизу), закиньте на GitHub.

2. Уявіть, що ваші зміни значно об'ємніші, ви міняли/добавляли/видаляли не по 1-2 строки, а десятки-сотні строк коду за кожен коміт. Тімлід ставить задачу: терміново закинути на продакшн ті зміни, які ви зробили в першому коміті (більший заголовок та синій колір дат). Використайте cherry-pick щоб закинути тільки цей коміт на продакшн.
